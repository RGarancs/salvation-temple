# Salvation Temple - Engineering Guide

## Project Overview
Church website for **Salvation Temple** (Храм Спасения) — an evangelical church in Riga, Latvia. Trilingual (RU/EN/LV), two themes (light cream / dark bordeaux).

**Domain**: `https://salvation-temple.lovable.app`

## Tech Stack
- **Framework**: React 18 + TypeScript + Vite 5
- **Styling**: Tailwind CSS 3.4 + custom design system in `src/index.css`
- **Routing**: React Router DOM 6 (6 pages + NotFound)
- **i18n**: Custom context-based system (`src/i18n/` JSON files + `LanguageContext.tsx`)
- **UI Components**: shadcn/ui (copy-paste, only `collapsible` and `tooltip` are used)
- **Fonts**: Montserrat (Google Fonts, loaded in index.css)
- **Path alias**: `@/` maps to `./src/`

## Commands
```bash
npm run dev          # Vite dev server on port 8080
npm run build        # Production build
npm run preview      # Preview production build on port 4173
npm run lint         # ESLint
npm run test         # Vitest
npx playwright test  # Visual regression tests (36 screenshots)
```

## Architecture

### Pages (`src/pages/`)
| Route | Page | Description |
|-------|------|-------------|
| `/` | Index.tsx | Home — hero, about, stats, ministries, events, contacts |
| `/training` | Training.tsx | Wholesome Life training program |
| `/history` | History.tsx | Church history timeline |
| `/donations` | Donations.tsx | Bank details + project cards |
| `/gallery` | Gallery.tsx | Photo masonry grid |
| `/testimonies` | Testimonies.tsx | Before/after testimonies |

### Section Components (`src/components/sections/`)
26 section components compose the pages. Each page wraps content in `<LanguageProvider>` + `<ChurchHeader>` + `<ChurchFooter>`.

### Shared Styles (`src/styles/bordeaux.ts`)
The dark bordeaux card pattern is used across 7 components. Import from `@/styles/bordeaux`:
- `bordeauxCardStyle` — CSS-in-JS background gradient
- `BordeauxOverlay` — texture overlay component (`src/components/ui/bordeaux-overlay.tsx`)
- `.bg-shine` CSS class — hover shine gradient overlay (replaces inline styles)

### i18n (`src/i18n/`)
- `ru.json`, `en.json`, `lv.json` — ~490 keys each
- `index.ts` — aggregates translations
- `LanguageContext.tsx` — React context provider (~40 lines)
- Translation key convention: `section.subsection.item` (e.g., `about.values.christ.title`)

## Design System

### Color Palette (CSS custom properties in `index.css`)
- **Light theme**: Warm cream background (`30 25% 97%`), chocolate foreground
- **Dark theme**: Bordeaux background (`350 35% 8%`), cream foreground
- **Extended palette**: cashmere, sunset, terracotta, amber, chocolate, coral, cream, burnt

### Gradient Text Classes
- `.text-gradient-sunset` / `.text-gradient-amber` / `.text-gradient-earth` / `.text-gradient-coral` / `.text-gradient-warm`
- Dark mode overrides exist for all gradient classes

### Card Patterns
- **Bordeaux cards**: Dark `hsl(350 35% 18%)` gradient + texture overlay. Used for premium content cards.
- **Glass cards**: `.glass`, `.glass-sunset`, `.glass-amber`, `.glass-coral`, `.glass-cashmere`
- **Warm cards**: `.card-warm`, `.card-glass`

### Spacing Convention (CSS utility classes in `index.css`)
- `.section-py` — responsive section padding (`py-8 md:py-12 lg:py-16`)
- `.page-py` — subpage top/bottom padding (`pt-28 pb-16`)
- `.section-container` — responsive container (`container mx-auto px-4 md:px-6 lg:px-8`)
- `.section-header` — centered header block (`text-center mb-10`)
- `.section-icon` — icon circle (`w-12 h-12 rounded-full mb-4`) — add bg gradient per-component
- `.section-title` — heading text (`font-display text-3xl md:text-4xl font-bold`) — add gradient + margin per-component

## Key Conventions
- All section components are named exports (`export const SectionName`)
- Page components are default exports wrapping `<LanguageProvider>`
- Gallery page uses `import.meta.glob('@/assets/gallery-*.jpg')` — new images auto-appear
- Other components import specific images via `@/assets/` path alias
- Bordeaux card styles imported from `@/styles/bordeaux`
- Route-level code splitting via `React.lazy()` + `<Suspense>`
- `ErrorBoundary` wraps the app root in `App.tsx`
- `loading="lazy"` on all images except hero slideshow and header logo

## Testing
- **Visual regression**: Playwright with 36 baselines (6 pages x 2 themes x 3 viewports)
- **Viewports**: Desktop 1280x720, iPad Mini 768x1024, iPhone 14 390x844
- **Theme toggle**: Click the ThemeToggle button (sun/moon icon in header)
- Run `npx playwright test` to verify no visual regressions
- Run `npx playwright test --update-snapshots` to update baselines after intentional changes

## Core Philosophy

- **Quality over speed.** There is no rush. Think deeply before writing code. A correct implementation now is always better than a fast one we fix later. The longer you think, the better.
- **Think before you act.** Before implementing anything, reason through the approach. Consider edge cases, existing patterns in the codebase, and potential conflicts. Second-guess yourself.
- **Push back when something is wrong.** If my request conflicts with existing architecture, introduces tech debt, has a better alternative, or is a bad practice — say so. Explain why and suggest the better path. If I insist after hearing your reasoning, proceed with my approach.
- **Never assume the stack.** Always scan the project first. Check `package.json`, config files, folder structure, existing patterns. The project tells you what tools and conventions to use.
- **Ask questions when uncertain.** If a task is ambiguous, has multiple valid approaches, or you're unsure about intent — ask before building. Don't guess.
- **Write code like a team of Google/Telegram engineers is reviewing it.** Not just working code — clean, scalable, navigable code. Proper separation of concerns, single-responsibility components, descriptive naming, no shortcuts.
- **Consistency beats preference.** Follow the patterns already established in the codebase. If the project uses a convention, use it — even if you'd prefer something else.
- **Rules have exceptions.** Every convention in this document exists for a reason, but none are absolute. If following a rule makes the code more complex, harder to understand, or over-engineered for the actual use case — flag it, explain why the exception is warranted, and proceed. The goal is good engineering judgment, not blind compliance. A convention broken with clear reasoning is better than a convention followed without understanding.

## Owner's Stack Profile

These are the default preferences for a frontend-focused fullstack developer — not hard rules. If a project's needs differ, adapt. But when starting fresh or making a choice, default to:

### Core Stack
- **Language:** TypeScript — strict mode, always. `strict: true`, `noImplicitAny: true`, `strictNullChecks: true`.
- **Mobile:** React Native via Expo (managed or bare).
- **Web:** Next.js (App Router, Server Components).
- **Monorepo:** npm workspaces for projects spanning web + mobile or multiple packages.

### State & Data
- **Client state:** Zustand with selectors. See *State Management* in Code Quality Standards.
- **Server state:** TanStack React Query v5. See *State Management* in Code Quality Standards.
- **Validation:** Zod schemas for all external data boundaries.

### Architecture
- **Feature-based folders.** Each feature module contains its own screens, hooks, stores, types, services, and components. Shared infrastructure lives in `core/` or `services/` or `hooks/`.
- **Single responsibility directories.** Types go in `types/`. Stores go in `stores/`. Services go in `services/`. No mixed-purpose folders.
- **Colocation when sensible.** Component-specific hooks, types, and utils live near the component. Shared ones go in shared directories.

### Patterns
- **ServiceResult pattern** for all service methods:
  ```typescript
  type ServiceResult<T> =
    | { success: true; data: T }
    | { success: false; error: string };
  ```
- **Custom hooks for all logic.** Components render UI. Hooks manage state, effects, and business logic. Never mix.
- **Static service classes** with try/catch returning ServiceResult. All errors logged via Logger, never swallowed.
- **Structured logging** via a Logger service. Never `console.log`. See `.claude/docs/logger-template.md` for the standard template.
- **Validated environment config** using Zod schemas. Never access `process.env` directly. See `.claude/docs/env-config.md`.

## Phased Execution

**Every task involving more than 2 files or a non-trivial feature MUST follow phased execution.**

### Phase 0 — Plan
- Break the task into logical phases.
- For each phase: what will be done, which files are affected, dependencies.
- Flag risks, unknowns, or areas needing clarification.
- **Before jumping into file structure, reason through the decisions.** Read `.claude/docs/decisions.md` for where state lives, what should be a service vs hook, and whether things belong in feature folders or shared.
- **Stop. Wait for approval.**

### Phase N — Implementation
- Execute one phase at a time.
- After each phase:
  - Summarize what was done.
  - List files created/modified.
  - Run quality gates.
  - Flag anything uncertain.
  - **Stop. Wait for approval.**

### Phase Final — Wrap-up
- Confirm all phases complete.
- Verify integration — does everything connect properly?
- Run all quality gates one final time.
- **Update ALL documentation** (ARCHITECTURE.md, README.md, /docs/patterns/, inline comments).
- Ask: "All good? Anything to adjust?"

### Small tasks (1-2 files, straightforward):
- No phased plan required.
- Still summarize what you did and check in after.
- Still run quality gates.
- Still update docs if relevant.

## Code Quality Standards

### TypeScript — Non-Negotiable
- **Strict mode always.** `strict: true`, `noImplicitAny: true`, `strictNullChecks: true`.
- **Zero `any` types.** Use `unknown` + type narrowing, or define proper interfaces. No exceptions.
- **Explicit return types** on all exported functions.
- **Interfaces for object shapes.** Use `type` for unions, intersections, and generics. Use `interface` for object contracts and anything that might be extended.
- **Zod schemas** for all external data boundaries — API responses, form inputs, database rows, configuration.

### Architecture
- Clean, scalable, readable architecture. Code that other developers navigate intuitively.
- Clear separation of concerns: components, hooks, utils, services, types, constants, stores.
- **Feature-based modules** as the default organizational pattern. Each feature owns its domain.
- Folder structure is project-dependent. Mirror and extend what already exists. If starting fresh, propose a structure based on the patterns described in this document and wait for approval.
- **No over-engineering.** Build what's needed now. YAGNI applies — if there's no second use case today, don't build the abstraction. See `.claude/docs/anti-patterns.md` for smell tests.
- **Flag architectural changes.** Never silently restructure or refactor architecture. Flag it, discuss it, then proceed.

### Component Philosophy
- **Single responsibility.** One component does one thing.
- **Reusability first.** Always ask: "Will this be used again?" If yes, extract it.
- **Custom hooks for logic.** Components render UI. Hooks manage logic. Never mix. If a component has more than 2 useState calls, extract to a hook.
- **No God components.** If a component file exceeds ~150 lines, it probably needs to be split.
- **Memoization by default.** `React.memo` on components. `useCallback` for handlers. `useMemo` for derived values, styles, and expensive computations.
- **Props over internal state** where it makes sense. Keep components predictable.
- **No inline objects/arrays in JSX.** Every render creates new references — this defeats `React.memo` and causes unnecessary re-renders down the tree. Extract to `useMemo` or module-level constants.

### Service Layer
- **Static service classes** with typed return values.
- **ServiceResult pattern** — every async service method returns `ServiceResult<T>`:
  ```typescript
  type ServiceResult<T> =
    | { success: true; data: T }
    | { success: false; error: string };
  ```
- **Every service method** wraps its body in try/catch. Errors go to Logger with context. Never swallow errors silently.
- **Never throw from services** unless it's a truly unrecoverable error. Return the error in the result type so callers handle it explicitly.

### State Management
- **Zustand** for client/UI state. Small, focused stores. One store per domain.
- **Always use selectors** to prevent unnecessary re-renders:
  ```typescript
  // WRONG
  const store = useAuthStore();

  // RIGHT
  const user = useAuthStore((s) => s.user);
  const isAuthenticated = useAuthStore((s) => s.isAuthenticated);
  ```
- **Actions live inside the store**, not in components.
- **TanStack React Query** for server/async state. Never raw `useEffect` for data fetching.
- **URL state** for filters, search, pagination — not Zustand.
- **Never fetch in `useEffect`.** Raw `useEffect` data fetching has no caching, no deduplication, no retry logic, and no offline support. Use TanStack React Query instead.

### Naming
- Descriptive, clear names. No abbreviations unless universally understood.
- Follow naming conventions already established in the project.
- Components: `PascalCase` — `RestaurantCard.tsx`
- Hooks: `camelCase` with `use` prefix — `useRestaurantLike.ts`
- Services: `PascalCase` — `RestaurantService.ts`
- Stores: `camelCase` with `Store` suffix — `authStore.ts`
- Utils: `camelCase` — `formatDate.ts`
- Types/Interfaces: `PascalCase` — `Restaurant`, `SessionConfig`
- Constants: `UPPER_SNAKE_CASE` — `MAX_RETRY_COUNT`
- Files: `camelCase.ts` for utils/hooks/stores, `PascalCase.tsx` for components

### Code Style
- Follow existing linting/formatting rules (ESLint, Prettier, Biome, etc.).
- If none exist, follow standard community conventions for the detected stack.
- One component per file. One hook per file. One service per file. One store per file.
- Avoid deep nesting. Extract early, return early.
- **Named exports only.** No default exports (exception: Next.js pages/layouts where required).
- **No barrel files (index.ts re-exports).** They break tree-shaking and obscure imports.

## Error Handling

- **Service methods** return `ServiceResult<T>`. Never throw unless truly unrecoverable.
- **Components** use Error Boundaries for crash recovery.
- **Every async operation** has a try/catch with meaningful error logging.
- **User-facing errors** are clear and actionable, not stack traces.
- **Loading states** present for every async operation — no blank screens.
- **Empty states** designed, not just hidden.
- **Never swallow errors.** Log them. Surface them. Handle them.

## Logging

- **Never use `console.log`.** Use the project's Logger service.
- **Structured logging** with context:
  ```typescript
  Logger.error('Failed to fetch sessions', error);
  Logger.info('[SessionService] Sync complete', { count: sessions.length });
  ```
- If no Logger service exists in the project, read `.claude/docs/logger-template.md` and create one before using console.log.

## Environment Configuration

- **Never access `process.env` directly.** Use a validated env config module. See `.claude/docs/env-config.md`.
- **Validate all env vars at startup** using Zod schemas. The app should fail fast with a clear error if config is missing or invalid.
- **Separate server and client schemas** in Next.js projects. Server secrets must never leak to the client bundle.
- **Maintain `.env.example`** with every variable and placeholder values. Commit it to git.

---

## Security

- **Run security reviews** on all code changes using `.claude/docs/security-reviewer.md`.
- **Never hardcode secrets.** API keys, tokens, and passwords go in environment variables.
- **Validate all external input** at the boundary with Zod. Never trust user input, URL params, or API responses without validation.
- **Use parameterized queries** for all database access. Never interpolate user input into queries.
- **Scope API permissions** to the minimum required. Don't use admin keys where read-only keys suffice.

## Testing

- **Framework:** Jest, Vitest, or whatever is already configured in the project.
- **Philosophy:** Lightweight and practical. Not overcomplicated.
- Test critical business logic, utility functions, complex hooks.
- Test component behavior, not implementation details.
- Don't test obvious things. Don't test library code.
- When building a new feature, include basic tests for core logic.
- When fixing a bug, add a test that would have caught it.
- **Service methods are the highest-priority test target** — they contain the business logic.
